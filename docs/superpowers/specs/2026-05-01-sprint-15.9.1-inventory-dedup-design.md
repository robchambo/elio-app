# Sprint 15.9.1 — Inventory Dedup — Design Spec

**Branch:** `sprint/15.9.1-inventory-dedup` (off `sprint/15.9-personalized-pantry` at HEAD `1fe34b7`)
**Status:** design spec — pending Rob review, then writing-plans hand-off
**Author:** Rob Thomas + Claude (sprint/15.9.1 brainstorm, 2026-05-01)
**Source material:**
- Audit findings during Sprint 15.9 final review: `FirestoreService.addInventoryItem` has zero dedup; every receipt scan, barcode scan, and builder-sheet inline custom-add creates a fresh Firestore doc.
- Existing `ShoppingService.addItem` matchKey + nameLower dedup pattern (the proven precedent we mirror).
- Brainstorm dialogue 2026-05-01 — tier policy = "existing tier sticks", expiry resets to "freshly purchased today" on re-scan, testing audience only (no legacy-user concerns).

## 1. Goal

When a user re-imports an item that's already in their pantry — via receipt scan, barcode scan, or any other write path — the existing inventory row updates rather than duplicating. The "purchase date" refreshes to today; for perishables the expiry refreshes to whatever the user picked on the new add; the existing tier sticks; `runningLow` clears; price refreshes if supplied. Caller-facing API is unchanged — `addInventoryItem(name, tier, ...)` still does the right thing, it just stops creating duplicates.

The success metric is concrete: scanning the same receipt twice produces N rows on the first scan and exactly N rows on the second scan, each with `lastPurchasedAt` updated to the second scan's timestamp.

## 2. Scope

### In scope

1. **Matching strategy** — primary `matchKey` (singularised lowercase), fallback `nameLower` (lowercase trim). Mirrors `ShoppingService.addItem` exactly. "Carrot" and "Carrots" merge into one row keyed by `"carrot"`.
2. **New inventory schema fields** — additive. `nameLower`, `matchKey`, `firstAddedAt`, `lastPurchasedAt`. Existing fields (`name`, `tier`, `category`, `expiryDate`, `price`, `runningLow`) unchanged.
3. **Refresh-on-re-add rule book** — tier sticky (existing wins always); expiry refreshes for perishables; `lastPurchasedAt` always refreshes; `firstAddedAt` never overwritten; `runningLow` clears; price replaces if supplied; `category` sticky.
4. **Migration of legacy inventory rows** — one-shot batched backfill on first inventory load post-15.9.1. Gated by a `inventoryDedupBackfilled: true` flag on the user doc (same pattern as `pantryMemoryBackfilled` from Sprint 15.9). Backfilled `firstAddedAt` and `lastPurchasedAt` set to `serverTimestamp()` at migration time — we don't know the real history, anchoring at "today" is harmless.
5. **Shared singularise utility** — `ShoppingService._singularise` extracted to a public utility (e.g. `PantryStringMatch.singularise` in `lib/utils/pantry_string_match.dart`) so the inventory dedup and shopping-list dedup consume one source of truth. Existing shopping tests stay green.

### Out of scope

| Deferred | Why |
|---|---|
| Tier-conflict UI prompt ("we noticed you scanned this as perishable but you had it as alwaysHave — change?") | Rule A picked: tier sticky, no auto-change. The user changes tier explicitly via long-press tier picker if they want. |
| Variant merging via `PantryUtils.normalise` (e.g. courgette ↔ zucchini) | Risks over-merging items the user might think of as distinct. Singularise-only is the conservative call. Variant maps can be a separate concern later. |
| `PantryUtils.normalise` integration in inventory matching | Same reason as above — keep matching simple. The variant map is for UI dedup warnings on add-pantry-item dialog, not for inventory keys. |
| On-device verification | Rob confirmed no test receipt available. Unit-test coverage of the dedup logic is the gate. |
| Legacy-user edge cases beyond happy path + idempotency | Testing audience only — no legacy-data concerns. |
| Tier upgrade heuristics (perishable → almostAlways → alwaysHave based on accumulated purchases) | Out of scope per Rule A. Could be a separate post-launch experiment. |

## 3. Architecture

### Data model

```
users/{uid}/inventory/{docId}     ← existing collection
  - name                           (existing) display name
  - tier                           (existing) 'alwaysHave' | 'almostAlwaysHave' | 'perishable'
  - category                       (existing, optional) PantryCategory.name
  - expiryDate                     (existing, perishables only) Timestamp
  - price                          (existing, optional) String
  - runningLow                     (existing) bool
  - nameLower                      (NEW) lowercase trim of `name` — fallback lookup
  - matchKey                       (NEW) singularised `nameLower` — primary lookup
  - firstAddedAt                   (NEW) server timestamp; sticky from first add
  - lastPurchasedAt                (NEW) server timestamp; refreshed on every write

users/{uid} (user doc)
  - inventoryDedupBackfilled       (NEW) bool flag, set true after first migration sweep
```

### New utility: `PantryStringMatch`

Pure-function utility for normalising pantry/shopping item names. Extracted from `ShoppingService._singularise` so both services consume the same logic.

```dart
// lib/utils/pantry_string_match.dart
class PantryStringMatch {
  PantryStringMatch._();

  /// Lowercase + trim. The primary fallback lookup key.
  static String nameLower(String name) => name.trim().toLowerCase();

  /// Singularised lowercase. The primary dedup key.
  /// Rules: skip <4 chars, skip ss/us/is endings, ies→y, oes/xes/ches/shes drop es,
  /// trailing s drop. Same logic as ShoppingService._singularise.
  static String matchKey(String name) { /* … */ }
}
```

`ShoppingService._singularise` is replaced by a delegation to `PantryStringMatch.matchKey`. Existing shopping tests confirm behaviour preservation.

### Refactored `FirestoreService.addInventoryItem`

The existing signature stays:

```dart
Future<String> addInventoryItem(
  String name, String tier, {
  DateTime? expiryDate,
  String? category,
  String? price,
}) async
```

New internal flow:

```
1. Compute lower = PantryStringMatch.nameLower(name)
2. Compute key   = PantryStringMatch.matchKey(name)
3. Query inventory where matchKey == key, limit 1
4. If no result, fallback query where nameLower == lower (legacy rows)
5. If existing row found:
     - Build update map per the rule book (tier sticky, expiry/runningLow/price refreshed, etc.)
     - Backfill missing fields (matchKey, nameLower, firstAddedAt) defensively
     - Always set lastPurchasedAt = serverTimestamp()
     - Update doc, return existing doc id
6. If no existing row:
     - Build full insert with all fields
     - firstAddedAt = lastPurchasedAt = serverTimestamp()
     - Insert, return new doc id
```

Errors propagate as before — no new error path needed.

### Storage abstraction (test seam)

Following the Sprint 15.9 pattern (`PantryMemoryStorage`), extract the Firestore-touching parts of `FirestoreService.addInventoryItem` into a small storage interface:

```dart
abstract class InventoryWriteStorage {
  /// Look up by matchKey first, falling back to nameLower for legacy rows.
  /// Returns the existing doc data + id, or null if no match.
  Future<({String id, Map<String, dynamic> data})?> findExistingByKey({
    required String matchKey,
    required String nameLower,
  });

  /// Update fields on an existing doc. The caller has already built
  /// the update map per the rule book.
  Future<void> updateInventoryDoc(String docId, Map<String, dynamic> updates);

  /// Insert a brand-new inventory doc. Returns the new doc id.
  Future<String> insertInventoryDoc(Map<String, dynamic> data);
}
```

`FirestoreService` retains the production wiring (private `_FirestoreInventoryWriteStorage` impl); a `FakeInventoryWriteStorage` lives in `test/fakes/` for the new unit tests.

### Migration: `InventoryMigrator`

A small one-shot service called **lazily from inside `addInventoryItem` itself** before the dedup query runs. This guarantees the migration has happened before any write that depends on the new fields, regardless of which surface the user reaches first (builder sheet, receipt scan, barcode scan, or "+ Add" chip).

Not exposed publicly; private to `FirestoreService` or a new `InventoryMigrator` class.

```
1. Read user doc; if inventoryDedupBackfilled == true, return immediately.
2. Read all inventory docs.
3. Build a Firestore batch.
4. For each doc missing matchKey:
     - Compute matchKey, nameLower from existing `name`.
     - Add fields + firstAddedAt + lastPurchasedAt = serverTimestamp().
     - batch.update(doc.ref, fields).
5. Set inventoryDedupBackfilled = true on user doc (in same batch).
6. Commit.
```

Idempotent — re-runs are no-ops via the flag, and the flag check is the first thing the migrator does so the cost on every subsequent `addInventoryItem` call is one user-doc read (or, with caching, zero — `FirestoreService` can keep a session-local boolean once the flag has been observed `true`).

To keep `addInventoryItem` performance reasonable on the happy path (flag already set), the in-memory cache matters: read the user doc once per session, remember `inventoryDedupBackfilled`, skip the migrator entirely on subsequent calls within the same session.

## 5. Refresh-on-re-add rule book

When `addInventoryItem` finds a matching existing row:

| Field | Behaviour |
|---|---|
| `tier` | **Sticky** — existing tier wins. Ignore the new arg. |
| `expiryDate` | **Replace with new value** if existing tier is `perishable` AND a new value is supplied. Else ignore. |
| `lastPurchasedAt` | **Always refresh** to `serverTimestamp()`. |
| `firstAddedAt` | **Never overwrite.** Backfill if missing on a legacy row, but never replace once set. |
| `runningLow` | **Reset to `false`** — fresh purchase clears the flag. |
| `price` | **Replace** if supplied; keep existing if not. |
| `category` | **Sticky** — keep existing. |
| `name` | **Sticky** — keep existing display casing/spelling. |
| `nameLower`, `matchKey` | **Replace** (defensive — should be unchanged but self-heals if backfill mismatched). |

When no existing row matches → insert with all fields, `firstAddedAt = lastPurchasedAt = serverTimestamp()`.

`addInventoryItem` returns the doc id either way (the existing one on update, the newly-created one on insert) — preserves the existing API contract.

## 7. Error handling

- **Storage errors** propagate up as exceptions (matches existing behaviour). The caller's existing try/catch (where present) handles them. No silent swallowing — failure to dedup IS a write failure worth surfacing.
- **Migration errors** silently no-op, like `PantryMemoryService.backfillFromInventoryIfNeeded` — the dedup logic itself doesn't depend on migration completing. Legacy rows without `matchKey` are caught by the `nameLower` fallback query forever.

## 8. Testing

- **`PantryStringMatch` unit tests** — 6-8 cases covering each plural rule + boundary + happy path. Existing `ShoppingService` tests stay green via the delegation.
- **`FirestoreService.addInventoryItem` dedup tests** (using `FakeInventoryWriteStorage`):
  - Insert path: no existing row → one insert with all 4 new fields populated.
  - Re-add same item → finds by `matchKey` → one update, no insert.
  - Re-add singular vs plural ("Carrots" then "Carrot") → merges to one row.
  - Tier-sticky rule: existing `alwaysHave`, re-add as `perishable` → tier stays `alwaysHave`, no expiry written.
  - Expiry refresh: existing `perishable` with old expiry, re-add with new expiry → expiry replaced, `runningLow` cleared.
  - Price refresh: re-add with new price → price replaced; re-add without price → price unchanged.
  - `firstAddedAt` is never overwritten on re-add.
  - Legacy fallback: row with only `nameLower` (no `matchKey`) is found and updated.
- **`InventoryMigrator` tests** — 2 tests: happy path (legacy rows get backfilled, flag set) + idempotency (flag set → no-op). Skip deep edge cases per the testing-audience-only framing.
- **Smoke** — `flutter analyze` clean; `flutter test` green.
- **No on-device verification** — Rob doesn't have a test receipt available. Unit-test coverage of the dedup logic is the gate.

## 10. Risks & open questions

- **Singularise rules are simple** — they don't catch irregular plurals (children, mice, etc.) or compound items ("baby spinach" vs "spinach"). Acceptable; the rules cover the bulk of grocery items and the worst case is a missed dedup (extra row), not data corruption.
- **`firstAddedAt` on legacy rows is "today" not the real first-add date.** Acceptable per Rob's testing-audience-only framing. For the analytics use ("you've had this 6 months") this matters less than getting dedup working correctly.
- **Race condition on first migration sweep.** If two app launches happen simultaneously (e.g. user has the app open on two devices), both might run the migration before either sets the flag. Not a correctness issue (idempotent — repeated backfills produce the same result) but produces duplicate writes. Acceptable; same race exists for `pantryMemoryBackfilled` and hasn't been a problem.
- **Scanner commit batches** call `addInventoryItem` per-item in a loop (`scanner_screen.dart:725`). Each call does a `findExistingByKey` round-trip — N items = 2N reads + N writes. For a 30-item receipt that's ~90 Firestore ops on commit. Acceptable for the scan-confirm workflow but worth noting; if performance becomes an issue we could add a batched-add API later.

## 11. Acceptance criteria

- [ ] Re-scanning the same receipt twice produces exactly N inventory rows (not 2N), each with `lastPurchasedAt` refreshed to the second scan's timestamp.
- [ ] Adding "Carrots" then "Carrot" (or vice versa) produces one row keyed by `matchKey: "carrot"`, with the first-typed name preserved as `name`.
- [ ] An existing `alwaysHave` item being re-imported via a scan that classifies it as `perishable` keeps `alwaysHave` and does NOT pick up an expiry.
- [ ] An existing `perishable` "Carrots" with last week's expiry, when re-scanned with a new "fresh" bucket pick, has `expiryDate` updated to the new value, `lastPurchasedAt` to today, and `runningLow` cleared.
- [ ] Re-adding an item without a price (e.g. via builder dialog) does NOT clear an existing price.
- [ ] On first builder-sheet open after upgrading to 15.9.1, legacy inventory rows get `matchKey`, `nameLower`, `firstAddedAt`, `lastPurchasedAt` populated. `inventoryDedupBackfilled: true` lands on the user doc.
- [ ] `flutter analyze` clean. `flutter test` green. Existing shopping-list dedup tests still pass.
- [ ] CLAUDE.md schema block reflects the four new inventory fields + the `inventoryDedupBackfilled` user-doc flag.
