## 1. Goal

Make the Pantry Builder feel personal by surfacing what each user has actually had before, using existing `tierMemory` + inventory data plus a new `customItems` set. Memory-driven, not predictive — every signal we surface is something the user has already demonstrated. No new top-level Firestore schema beyond a single new sub-collection.

The success metric is qualitative: "the builder feels like mine, not a generic catalogue." Concretely, a returning user opens the builder and the first 20 items they see are the items they've actually had in their pantry, ordered by recency. Their typed customs reappear automatically. Items they've eaten before are visually marked inside the canonical category list so the familiar landscape is scannable.

## 2. Scope

### In scope

1. **"Your usuals" section** — top-of-builder list, sourced from `users/{uid}/tierMemory`, ordered by `lastSeen` desc, capped at 20 items. Hidden when empty (graceful for new users). Items render as `_BuilderChip`s identical to category items, so the existing tap-to-add / long-press-to-pick-tier behaviour applies unchanged.
2. **"Had before" markers** — items inside categories that already exist in `tierMemory` get a subtle dot (or tint) next to the label so familiar items stand out from the canonical list. Visual noise must stay quiet — this is a hint, not a CTA.
3. **Custom-item persistence** — when a user types via "+ Add custom item" and picks a tier, the entry writes to a new `users/{uid}/customItems/{normalizedName}` doc with `displayName`, `category`, `tier`, `firstSeen`, `lastSeen`. On subsequent opens, customs surface as first-class chips inside the category they were saved under (with a small "yours" indicator) AND in "Your usuals" if they qualify by recency.
4. **Backfill `tierMemory` from current inventory** — one-shot migration on first builder open after this lands. Existing users see "Your usuals" populated immediately without needing to re-scan a receipt.
5. **`DietaryFilter` extension to the Pantry Builder sheet** — the existing dietary-conflict greying that runs on onboarding screens 11/12 (`lib/utils/dietary_filter.dart` → `_blockReasons`) extends to the in-app builder. Greys out items that conflict with the user's current dietary settings inside both category lists AND "Your usuals". Item-level only — categories stay visible (a vegetarian who eats fish should still see the Mediterranean section).
6. **Tier defaulting in "Your usuals"** — tapping an item from "Your usuals" defaults to the tier stored on its `tierMemory` row, not the category-default `alwaysHave`. Long-press still opens the tier picker for override. This is the small UX detail that makes the section feel like memory, not a list of buttons.
7. **Universal staples are excluded from the builder entirely** — salt, pepper, water, sugar, and generic cooking oil (cooking oil / vegetable oil / sunflower oil / canola oil / rapeseed oil / neutral oil / generic oil) don't appear in any category list, "Your usuals", or "Had before" markers, and can't be persisted as customs. **Specific flavour oils stay in scope** (sesame oil, olive oil, chilli oil, truffle oil, etc. — these are recipe-defining, not interchangeable). The exact set is already implemented in `ShoppingService._stapleWords` + `_genericOilsExact` and used via `isStaplePublic(normalisedName)` — Task 7 extracts these to a shared `PantryStaples` utility class so both services consume the same source of truth. Typing one as a custom shows a friendly inline note ("salt is always assumed — no need to add it") and the dialog Cancel button. Backfill skips them when migrating from inventory.
8. **On-device design review beat** — after Tasks 1–5 land, do an explicit on-device pass before declaring the sprint done. The builder now stacks three indicator systems on top of each other: had-before dots, customs eyebrow, dietary-greyed chips. Visual review the cumulative density and pull back any indicator that doesn't earn its weight (most likely candidates if soup happens: drop the "yours" eyebrow and just leave customs as first-position chips, or drop the dot in favour of bold-only on had-before items). Allocate 30 minutes for this beat; treat as a real review gate, not a checkbox.

### Out of scope (explicitly deferred)

| Deferred | Why |
|---|---|
| AI-generated starter pantry from dietary + cuisine + region | High risk of wrong recommendations on a first-impression surface. Latency + cost on onboarding. Defer until usage data shows cold-start gap matters. |
| Smart category ordering by inferred cuisine usage | "Your usuals" already serves this need with higher precision (actual items > inferred preferences). Reordering 12 categories is small UX win for big code. |
| "Suggested for you" carousel on the Pantry tab | Yet another surface. Had-before-but-not-in-pantry signal already lives inside the builder via "Your usuals". Don't fight for tab real estate. |
| Hide whole categories that conflict with dietary | Wrong granularity — item-level filtering (#5) is correct. Hiding sections confuses users who recognise category names. |

## 3. Architecture

### Data sources

```
users/{uid}/tierMemory/{normalizedName}     ← already exists (scanner-populated)
  - name (display)                           Memory-of-tier per item.
  - tier ('alwaysHave' | 'almostAlwaysHave' | 'perishable')
  - lastSeen (Timestamp)
  - normalizedName (lowercased + trimmed key)

users/{uid}/customItems/{normalizedName}    ← NEW sub-collection (this sprint)
  - displayName
  - category   (PantryCategory.name string)
  - tier       (chosen at add-time; remembered)
  - firstSeen
  - lastSeen
  - normalizedName

users/{uid}/inventory/{id}                  ← already exists (current pantry)
  Used for backfill (#4) and the existing in-pantry indicator on chips.
```

### New service: `PantryMemoryService`

Pure data-access layer for the new behaviours. Lives at `lib/services/pantry_memory_service.dart`. Single instance, accessed via `instance` getter (matches `ShoppingService` pattern).

```dart
class PantryMemoryService {
  static final instance = PantryMemoryService._();
  PantryMemoryService._();

  /// Top N items from tierMemory ordered by lastSeen desc.
  Future<List<PantryMemoryEntry>> recentUsuals({int limit = 20});

  /// Set of normalised names that exist in tierMemory — for the
  /// "had before" marker on category items. Returns a Set so the chip
  /// builder can do a single O(1) lookup per item.
  Future<Set<String>> hadBeforeKeys();

  /// User's persisted custom items. Returned as a category → list map
  /// so the Pantry Builder sheet can render them as first-class chips
  /// inside the right category section.
  Future<Map<String, List<PantryMemoryEntry>>> customsByCategory();

  /// Persist a typed custom item. Idempotent on the normalisedName
  /// key — re-typing "Miso paste" updates lastSeen + tier, doesn't
  /// duplicate.
  Future<void> upsertCustom({
    required String displayName,
    required String category,
    required String tier,
  });

  /// One-shot migration: walks current inventory and writes any
  /// missing tierMemory rows, using inventory tier as the seed. Idempotent
  /// — safe to call multiple times. Driven by a flag on the user doc
  /// (`subscription.tierMemoryBackfilled`) so we don't re-run on every
  /// open.
  Future<void> backfillFromInventoryIfNeeded();
}
```

### Builder sheet integration

`lib/widgets/pantry_builder_sheet.dart` gains:

- An `initState` call to `PantryMemoryService.backfillFromInventoryIfNeeded()` (fire-and-forget, error-swallowed).
- Three async loads in parallel on first build: `recentUsuals()`, `hadBeforeKeys()`, `customsByCategory()`. Stored in state, re-run on rebuild only if a custom is added.
- A new "Your usuals" `Column` rendered above the category `ListView` when the list is non-empty. Same `_BuilderChip` widget — only difference is the chip's `onTap` calls a new `_addFromUsuals(entry)` that uses `entry.tier` as the default instead of `'alwaysHave'`.
- Each category's chip wrap gets a leading-dot indicator (creamDeep filled circle, ~6px) on items whose `normalizedName` is in `hadBeforeKeys()`.
- Custom items render as the first chips inside their saved category, with a small "yours" eyebrow above them, then the canonical items follow.
- Existing `showAddPantryItemDialog` flow extended: when user picks a tier for a typed custom, the dialog also calls `PantryMemoryService.upsertCustom(...)` before returning the result.
- New `DietaryFilter.blockReasons` call on every chip; when non-empty, the chip renders in the existing greyed-out style (already implemented for onboarding screen 11/12 — copy the styling).

## 5. Data flow on first builder open

```
showPantryBuilder() called
  └─ PantryBuilderSheet.initState()
       ├─ fire-and-forget: backfillFromInventoryIfNeeded()
       │     └─ if subscription.tierMemoryBackfilled != true:
       │           - read users/{uid}/inventory
       │           - for each missing row: write users/{uid}/tierMemory/{name}
       │           - set subscription.tierMemoryBackfilled = true
       └─ Future.wait([
              recentUsuals(limit: 20),
              hadBeforeKeys(),
              customsByCategory(),
            ])
            └─ setState((): _usuals/_hadBefore/_customs)
                 └─ build() renders:
                     [Search field]
                     [Custom add row]
                     [Your usuals]      ← new, hidden if empty
                     [Categories]
                       └─ each category:
                           - "yours" customs first (first-class chips)
                           - canonical items, with had-before dots
                             and dietary-greyed if conflicting
```

A user adding a custom item:
```
+ Add custom item → showAddPantryItemDialog → tier picker
  ├─ existing onAddItem callback (writes inventory + tierMemory via the existing scanner path)
  └─ NEW: PantryMemoryService.upsertCustom(displayName, category, tier)
        └─ writes customItems/{normalizedName}
```
On the next builder open the typed custom appears in `customsByCategory()` AND in `recentUsuals()` (if new enough).

## 6. Error handling

- All `PantryMemoryService` reads return empty collections on error rather than throwing (matches existing `getUserData()` pattern in the codebase). The builder UX gracefully degrades to "no usuals yet" / "no had-before markers" rather than blowing up.
- Backfill silently no-ops on read failure (don't block the builder for an optional polish feature).
- `upsertCustom` writes are fire-and-forget with a logged error on failure — the user's custom still lands in inventory through the existing `onAddItem` callback, just not in the persistent customs list.

## 9. Risks & open questions

- **Backfill cost** — a user with 200 inventory items gets 200 tierMemory writes on first open. Single batched write keeps it to one round trip; should be fine but worth verifying on a populated test account.
- **Custom-item normalisation collisions** — `PantryUtils.normalise()` strips plurals + variants. "Carrot" and "Carrots" share a key — we keep the first-typed `displayName`. Document this.
- **Custom in two categories** — if a user types a name and picks one category once, then later types it again and picks a different category, the second upsert wins. Acceptable: the user expressed a preference. No multi-category support.
- **Chip noise from dots + custom-eyebrow + dietary-greyed** — addressed by Task 8 (on-device design review beat). Concrete pull-back candidates if the layout reads soup: drop the "yours" eyebrow (customs land in first chip position regardless) or drop the dot in favour of bold-only on had-before items.
- **Universal-staples list scope** — start with the same set the shopping-list filter uses (salt, pepper, water — verify in `ShoppingService.isStaplePublic`). If we add others (oil? sugar?), we risk hiding items some users genuinely want to track. Keep the list conservative; add to it via a deliberate decision later, not by default.

## 10. Acceptance criteria

- [ ] A returning user with non-empty `tierMemory` opens the builder and sees "Your usuals" at the top with their N most-recent items, ordered by recency.
- [ ] Tapping an item from "Your usuals" adds it to inventory at the tier remembered in `tierMemory`, not `alwaysHave`.
- [ ] Items inside a category that exist in `tierMemory` show a subtle leading dot.
- [ ] A user types a custom "Miso paste" → picks "Always have" → it persists. Re-opening the builder shows "Miso paste" as a first-class chip inside whichever category was chosen, with a "yours" indicator.
- [ ] An existing user (pre-15.9 install) opens the builder for the first time post-update and sees "Your usuals" populated by the backfill — no scan required.
- [ ] A vegan user sees `Dairy & Eggs` items greyed out with "Vegan" as the reason chip; they remain visible (not hidden).
- [ ] Salt, pepper, water, sugar, and generic cooking oil (cooking oil / vegetable oil / sunflower / canola / rapeseed / neutral / generic oil) do NOT appear in any category list, "Your usuals", or "Had before" markers. Typing any of them as a custom shows the inline note instead of persisting. **Specific flavour oils remain available** — sesame oil, olive oil, chilli oil, truffle oil etc. continue to render as normal items.
- [ ] On-device design review pass completed; any indicator pull-back decisions captured in commit messages.
- [ ] `flutter analyze` clean. `flutter test` green. APK builds and runs.
