## 1. Goal

Let two adults — each with their own Firebase Auth account — share a household's inventory, shopping list, meal plan, customs, scanner learning, and dietary union, while keeping their personal data (saved recipes, ratings, fcmTokens, subscription) private to them. The existing "profiles[] for dependents without phones" model lives on alongside the new auth-user-based members, so kids and partners without an Elio account stay in scope.

Three concrete success criteria:

1. Rob signs in on his phone, creates a household, invites Kate via 6-digit code. Kate enters the code on her own Elio install. From that moment, edits to inventory/shopping/meal-plan on either phone appear on the other in real time. Their dietary needs are union-merged into Gemini's recipe-generation prompt.
2. Rob is Elio Pro; Kate isn't. Once Kate joins the household, her app sees Pro features unlocked (meal planner, shopping list, scanning, recipe import). She's covered for as long as she's in the household.
3. Existing solo users see **no behavioural change** until they explicitly create or join a household. The opt-in layer is invisible until they engage it.

## 2. Locked design decisions (from 16.7a brainstorm)

| # | Question | Decision | Implication |
|---|---|---|---|
| 1 | How "shared" is a household? | Full share with opt-in (no force-migrate) | Inventory, shopping list, meal plan, customItems, tierMemory, dietaryUnion all move to `households/{hid}/*`. Per-user data (recipes, ratings, fcmTokens, subscription, user-doc dietary) stays at `users/{uid}/*`. Solo users untouched. |
| 2 | Migration UX when a new member joins? | Owner seeds; invitee chooses via explicit "Import my pantry" action | Owner's per-UID shared data is **copied** into the household at creation. Invitee joins empty and decides whether to import their stuff (deduped via existing `InventoryWriter` from Sprint 15.9.1). |
| 3 | Invite mechanism? | 6-digit numeric code, 24h TTL, single-use | No domain dependency, no Cloud Functions required for v1. Top-level `inviteCodes/{code}` doc. |
| 4 | Pro entitlement on the household? | Owner's Pro extends to all members | No new RC SKU. Beats AnyList household price by extending existing $29.99/yr to up to 6 members (effective $5/user/year). |
| 5 | Where does dietary live in multi-UID? | Per-user dietary on user doc (NEW fields), union cached on household doc | Each adult owns their own allergens — they travel with them when they leave the household. Cached union refreshed on every dietary edit. Position-1 allergen preamble in Gemini prompt reads the cache. **Today's code keeps dietary at `users/{uid}/profiles/{owner}.dietaryRequirements` + `.allergies` — see §8.0 for the one-time migration to new user-doc fields `dietary` + `allergens`.** |
| 6 | Ownership lifecycle? | Single owner. Kick + leave allowed. Owner cannot leave (must delete). Owner deletes account → 7-day cascade-delete warning to members. No co-owners, no handoff in v1. | |

Two additional decisions captured in §3-§4 brainstorm:
- **Gate household creation behind Pro.** Free users see the tile but a Pro paywall fires on tap.
- **Sprint 17 dependencies** for the RC webhook + monthly sweep (§9.3) — both close the client-side cheating hole and the offline-grace edge.

## 3. Architecture

### 3.1 Two-kind member model

| Type | Where it lives | What they own |
|---|---|---|
| **Auth member** (adult with own Elio account) | UID in `households/{hid}.members[]` | Their own user doc, dietary, recipes, ratings, fcmTokens, subscription. |
| **Profile** (dependent without own phone — current model) | Entry in `households/{hid}/profiles/{id}` | A name + dietary requirements + `addedBy` UID. Edited by any household member (owner or peer). |

Auth members and profiles together compose the dietary union (§5.2). Existing solo users keep using `users/{uid}/profiles/*` exactly as today.

### 3.2 Opt-in households as a new layer

A new field on the user doc — `householdId: string?` — gates which data path the repositories use:

- `householdId == null` (default) → solo. Reads/writes shared data at `users/{uid}/{collection}/`. **This is every user today.**
- `householdId == "hid"` → household member. Reads/writes shared data at `households/{hid}/{collection}/`.

The old per-UID subcollections are **never deleted** on join. Leaving the household reverts the user to their old per-UID data — whatever was there pre-join is intact.

### 3.3 Repository abstraction — `SharedDataScope`

A new layer that resolves the right Firestore path. Every screen that reads/writes shared data calls into it instead of hardcoding `users/{uid}/`. The three repositories that need updating:

1. `FirestoreService` (used everywhere — pantry, meal plan, shopping)
2. `InventoryWriter` (Sprint 15.9.1 dedup writer)
3. `ShoppingService` (matchKey dedup writer)

```dart
// New abstraction
class SharedDataScope {
  static Future<DocumentReference> root() async {
    final uid = FirebaseAuth.instance.currentUser!.uid;
    final user = await FirebaseFirestore.instance.collection('users').doc(uid).get();
    final hid = user.data()?['householdId'] as String?;
    return hid == null
      ? FirebaseFirestore.instance.collection('users').doc(uid)
      : FirebaseFirestore.instance.collection('households').doc(hid);
  }
}
```

Repositories take the resolved ref and append the subcollection name. **Screens don't change** — they keep calling `FirestoreService.getInventory()` etc.; the service handles routing.

Caching note: the resolved scope can be cached for the session (refresh on `householdId` change events — leaving, kicking, etc.). One extra Firestore read at session start is acceptable.

## 4. Firestore schema

### 4.1 User doc — one new field

```
users/{uid}
  (user doc) — existing fields unchanged
    + householdId: string?       (null = solo; otherwise points to households/{hid})
```

Everything else unchanged. Dietary stays on user doc. Subcollections (`recipes/`, `ratings/`, `fcmTokens/`, `subscription`) stay per-user.

### 4.2 New top-level `households/{hid}/` subtree

```
households/{hid}/
  (household doc):
    ownerUid:              string              // creator; immutable after create
    members:               array<uid>          // includes owner; cap 6
    name:                  string              // optional household label (default "Our household")
    proActive:             bool                // mirror of owner's RC isPro
    proActiveSyncedAt:     timestamp           // when owner's app last wrote proActive
    deletionScheduledAt:   timestamp?          // set when owner deletes account; 7d countdown
    createdAt:             timestamp

  profiles/{profileId}                         // non-auth dependents (kids, partner without account)
    name:                  string
    dietaryRequirements:   array<string>
    addedBy:               uid
    createdAt:             timestamp

  dietaryUnion/singleton                       // cached union of all members + profiles
    dietary:               array<string>
    allergens:             array<string>
    updatedAt:             timestamp
    computedFrom:          { memberUids: array<uid>, profileIds: array<string> }

  inventory/{itemId}                           // schema identical to users/{uid}/inventory/*
  shoppingItems/{itemId}                       // schema identical to users/{uid}/shoppingItems/*
  mealPlan/{docId}                             // schema identical to users/{uid}/mealPlan/*
  customItems/{normalizedName}                 // schema identical to users/{uid}/customItems/*
  tierMemory/{normalizedName}                  // schema identical to users/{uid}/tierMemory/*
```

The shared subcollections (`inventory/`, `shoppingItems/`, `mealPlan/`, `customItems/`, `tierMemory/`) have **identical document schemas** to their per-UID counterparts. The migration on household creation is a pure copy — no schema transformation needed.

### 4.3 Top-level `inviteCodes/{code}` subtree

```
inviteCodes/{code}                             // doc ID is the 6-digit code
  hid:                   string                // target household
  ownerUid:              string                // creator; audit only
  expiresAt:             timestamp             // 24h after creation
  used:                  bool                  // single-use flag
  usedByUid:             uid?                  // filled on accept
  createdAt:             timestamp
```

Top-level (not under household) because the invitee doesn't know `hid` until they accept — they only have the code. They look up `inviteCodes/{code}` directly. Rules allow signed-in `get` by exact ID but not `list`, so codes can't be enumerated.

Keyspace math: 6 digits = 1,000,000 codes. With ~100 active codes at any time, a random guess has a ~0.01% hit rate. Firestore's default per-IP sustained read rate (effectively ~1 read/sec without App Check) means a brute-force attacker would average ~1 successful guess per ~3 hours of continuous attempts on a single household. **Mitigations layered:**
- Single-use flag closes the door immediately after first claim.
- 24h TTL bounds the window per code.
- App Check enforcement (Sprint 17) raises the per-app cost of guessing significantly.
- Rule rejects update unless `usedByUid == request.auth.uid` (an attacker can't claim a code on behalf of someone else).

Acceptable for v1; revisit if abuse signals appear post-launch.

### 4.4 What stays per-user (intentional)

| Subcollection | Why per-user |
|---|---|
| `users/{uid}/recipes/` | Saved recipes are your taste profile. Sharing dilutes personalisation. |
| `users/{uid}/ratings/` | Like/dislike feeds your adaptive learning. |
| `users/{uid}/fcmTokens/` | Device-level. |
| `users/{uid}.subscription` | RC is per-UID. Shared entitlement via `households/{hid}.proActive` cache. |
| `users/{uid}/profiles/` (legacy) | Stays so solo-user code path is untouched. Dormant when user is in a household. Reactivates on leave. **Never deleted.** |
| `users/{uid}.dietary` (on user doc) | Each adult owns their dietary; allergens travel with them when they leave. |

## 5. Security rules

### 5.0 Baseline gap to fix in 16.7b (independent of households)

The current `firestore.rules` (as of `8fbc553`) lists owner-only rules for `profiles/`, `inventory/`, `recipes/`, `ratings/`, `mealPlan/`, `shoppingItems/`, `tierMemory/`, `fcmTokens/` — but **not `customItems/`**. The Sprint 15.9 `customItems` writes (`lib/services/pantry_memory_service.dart`) have been going to a collection that the default-deny rule technically blocks. Worth Rob verifying current behaviour in prod (if writes are succeeding, something else is going on; if they're failing, this is a latent bug). Either way, 16.7b adds the `customItems/{id}` rule both for the per-UID and the household subtrees.

### 5.1 Sketch (final form in §10 file touchpoints)

```javascript
match /users/{uid} {
  // EXISTING rules unchanged.
  allow read:   if isOwner(uid);
  allow create: if isOwner(uid) && initialSubscriptionSafe();
  allow update: if isOwner(uid) && protectedSubKeysUnchanged();
  allow delete: if isOwner(uid);

  match /profiles/{id}      { allow read, write: if isOwner(uid); }
  match /inventory/{id}     { allow read, write: if isOwner(uid); }
  match /recipes/{id}       { allow read, write: if isOwner(uid); }
  match /ratings/{id}       { allow read, write: if isOwner(uid); }
  match /mealPlan/{id}      { allow read, write: if isOwner(uid); }
  match /shoppingItems/{id} { allow read, write: if isOwner(uid); }
  match /tierMemory/{id}    { allow read, write: if isOwner(uid); }
  match /customItems/{id}   { allow read, write: if isOwner(uid); }
  match /fcmTokens/{id}     { allow read, write: if isOwner(uid); }
}

match /households/{hid} {
  function isMember() {
    return isSignedIn()
      && request.auth.uid in resource.data.members;
  }
  function isOwnerOfHousehold() {
    return isSignedIn()
      && request.auth.uid == resource.data.ownerUid;
  }
  function proActiveUnchanged() {
    // Post-Sprint-17: enforces Admin-SDK-only writes to proActive.
    // Sprint 16.7b ships with this rule COMMENTED OUT — owner writes it client-side.
    // Sprint 17 Cloud Function lands → flip this on.
    return request.resource.data.proActive == resource.data.proActive;
  }

  // Helper for the invitee-self-add path: verify the requester has a valid
  // used-and-claimed-by-self invite code targeting this household.
  function hasValidInviteFor(hid) {
    let codeId = request.resource.data.get('joinedVia', '');
    return codeId != ''
      && exists(/databases/$(database)/documents/inviteCodes/$(codeId))
      && get(/databases/$(database)/documents/inviteCodes/$(codeId)).data.hid == hid
      && get(/databases/$(database)/documents/inviteCodes/$(codeId)).data.used == true
      && get(/databases/$(database)/documents/inviteCodes/$(codeId)).data.usedByUid == request.auth.uid
      && get(/databases/$(database)/documents/inviteCodes/$(codeId)).data.expiresAt > request.time;
  }

  allow read:   if isMember();
  allow create: if isSignedIn()
                 && request.resource.data.ownerUid == request.auth.uid
                 && request.auth.uid in request.resource.data.members
                 && request.resource.data.members.size() == 1;
  // Two update paths: (a) existing member updates non-membership fields;
  // (b) invitee self-adds via valid invite code.
  allow update: if
    // (a) member-driven update (owner kicks, rename, etc.)
    (
      isMember()
      && request.resource.data.ownerUid == resource.data.ownerUid
      && request.resource.data.members.size() <= 6
    )
    ||
    // (b) invitee joining via valid used invite code
    (
      isSignedIn()
      && !(request.auth.uid in resource.data.members)
      && request.auth.uid in request.resource.data.members
      && request.resource.data.members.size() == resource.data.members.size() + 1
      && request.resource.data.members.hasAll(resource.data.members)
      && request.resource.data.ownerUid == resource.data.ownerUid
      && request.resource.data.members.size() <= 6
      && hasValidInviteFor(hid)
      // Note: client passes the redeemed code as `joinedVia` in the update
      // payload (transient field — server-side write can clear or persist it).
      // && proActiveUnchanged() — add in Sprint 17
    );
  allow delete: if isOwnerOfHousehold();

  match /profiles/{id}      { allow read, write: if isMember(); }
  match /inventory/{id}     { allow read, write: if isMember(); }
  match /shoppingItems/{id} { allow read, write: if isMember(); }
  match /mealPlan/{id}      { allow read, write: if isMember(); }
  match /customItems/{id}   { allow read, write: if isMember(); }
  match /tierMemory/{id}    { allow read, write: if isMember(); }
  match /dietaryUnion/{id}  { allow read, write: if isMember(); }
}

match /inviteCodes/{code} {
  // Read by exact code only — no listing/querying.
  allow read:   if isSignedIn() && resource != null;  // single-doc get; list is implicitly off without an index
  allow list:   if false;
  // Owner creates the code; rule cross-references the household.
  allow create: if isSignedIn()
                 && request.resource.data.ownerUid == request.auth.uid
                 && get(/databases/$(database)/documents/households/$(request.resource.data.hid))
                     .data.ownerUid == request.auth.uid;
  // Invitee marks used on accept. Tight constraints to prevent code mutation
  // (e.g. retargeting hid, extending TTL, claiming on behalf of someone else).
  allow update: if isSignedIn()
                 && resource.data.used == false
                 && request.resource.data.used == true
                 && request.resource.data.usedByUid == request.auth.uid
                 && request.resource.data.hid == resource.data.hid
                 && request.resource.data.ownerUid == resource.data.ownerUid
                 && request.resource.data.expiresAt == resource.data.expiresAt
                 && request.resource.data.createdAt == resource.data.createdAt;
  allow delete: if isSignedIn() && resource.data.ownerUid == request.auth.uid;
}

match /{document=**} { allow read, write: if false; }
```

### 5.2 Key invariants

1. **Owner is immutable after creation.** No client-side handoff in v1.
2. **Members cap at 6.** Enforced server-side.
3. **Invite codes are single-use.** `update` requires `resource.data.used == false`.
4. **Cross-UID writes don't happen.** All cross-user collaboration goes through `households/{hid}/`, which all members can access. Members never directly touch each other's user docs.
5. **Member self-reset on permission-denied.** When kicked or when a household is deleted by the owner, the affected user's `householdId` stays stale on their user doc until their app detects a permission-denied read of `households/{hid}/` and self-resets. Eventual consistency, acceptable.

## 6. Lifecycle flows

Eight flows. Each described as trigger → operations → end state. Edge cases captured inline.

### 6.1 Create household (owner)

**Trigger:** Solo user (no `householdId`) on AccountScreen taps "Create a household". If Free → paywall fires (Pro-gated per §2). If Pro → confirm dialog with copy "Your inventory, shopping list, and meal plan will be shared with anyone you invite."

**Operations** (best-effort batched; partial-failure path acceptable — a single Firestore transaction cannot span >500 doc writes or arbitrary subcollections, so the implementation uses idempotent batches with retry, not one atomic transaction):
1. Generate `hid` (auto-ID).
2. Create `households/{hid}` doc: `ownerUid`, `members: [ownerUid]`, `proActive` mirrored from current RC state, `proActiveSyncedAt`, `createdAt`.
3. **§8.0 dietary migration first:** read `users/{ownerUid}/profiles/{ownerProfileId}.dietaryRequirements` + `.allergies`, write to new user-doc fields `users/{ownerUid}.dietary` + `users/{ownerUid}.allergens`. Idempotent — skips if user-doc fields already exist.
4. **Copy** owner's shared data into household subtree:
   - `users/{ownerUid}/inventory/*` → `households/{hid}/inventory/*`
   - `users/{ownerUid}/shoppingItems/*` → `households/{hid}/shoppingItems/*`
   - `users/{ownerUid}/mealPlan/*` → `households/{hid}/mealPlan/*` (today: `mealPlan/current` only; spec routes to household scope from here, owner's per-UID `users/{uid}/mealPlan/current` stays in place but is unreachable by the app while in household — see §6.4 step 4 note)
   - `users/{ownerUid}/customItems/*` → `households/{hid}/customItems/*`
   - `users/{ownerUid}/tierMemory/*` → `households/{hid}/tierMemory/*`
   - `users/{ownerUid}/profiles/*` (**all** profiles including `isOwner: true` — its dietary is already mirrored to user-doc per step 3, but keeping the profile doc preserves the existing dietary-screen UX during the §8.0 transition window) → `households/{hid}/profiles/*`
5. Compute and write `households/{hid}/dietaryUnion/singleton` from owner's user-doc `dietary` + `allergens` (post step 3 migration) ∪ all profile `dietaryRequirements`.
6. Write `users/{ownerUid}.householdId = hid`.
7. **Source per-UID subcollections are kept** — not deleted. Leaving reverts cleanly.

**Failure recovery:** if step 2-5 succeeds but step 6 fails, the household exists but no user references it. The owner's app sees them as still-solo on next refresh. A daily Cloud Function (Sprint 17) sweeps orphan households where no user has the matching `householdId`. Not blocking for v1 — manual cleanup acceptable.

### 6.2 Generate invite code (owner)

**Trigger:** Owner in HouseholdScreen taps "Invite member".

**Operations:**
1. Generate 6-digit code (uniform random, no patterns).
2. Create `inviteCodes/{code}` with `hid`, `ownerUid`, `expiresAt = now + 24h`, `used: false`. On collision (~0.01% per attempt with 100 active codes): retry up to 3 times.
3. Show code in a copyable card with "Share" button (system share sheet: "Join my Elio household with code: 123456. Expires in 24h.").

**Multiple codes per household:** allowed. Old codes stay valid until their 24h TTL or owner manually revokes. No "revoke all". Cloud Function sweep purges expired codes; optional, not blocking.

### 6.3 Accept invite (invitee)

**Trigger:** Signed-in user on AccountScreen taps "Join a household" → dialog with code input → tap Join.

**Operations** (sequenced — Firestore transactions can't span all the cross-doc writes cleanly, so the order matters for rule satisfaction):
1. Read `inviteCodes/{code}`. Reject if missing, `used == true`, or `expiresAt < now`. Surface friendly error.
2. Read `households/{hid}` to check `members.size() < 6`. Reject if full ("This household is full.").
3. Reject if invitee already has `householdId` ("You're already in a household. Leave it first.").
4. **§8.0 dietary migration:** read invitee's `users/{inviteeUid}/profiles/{ownerProfileId}.dietaryRequirements` + `.allergies`, write to `users/{inviteeUid}.dietary` + `users/{inviteeUid}.allergens`. Idempotent.
5. Update `inviteCodes/{code}.used = true`, `usedByUid = invitee.uid`. (This must precede step 6 — the household rule allowing a non-member self-add references this used-and-claimed-by-self state. See §5.1 invitee-self-add rule branch.)
6. `arrayUnion(invitee.uid)` on `households/{hid}.members`. Rule path: invitee is not a member yet, but the household update rule has a branch that accepts the write when (a) the size increase is exactly one, (b) the added UID is `request.auth.uid`, and (c) there's a matching `inviteCodes/{code}` doc with `used: true` and `usedByUid == request.auth.uid` (§5.1 below).
7. Update `users/{inviteeUid}.householdId = hid`.
8. Recompute `households/{hid}/dietaryUnion/singleton` (adds invitee's now-on-user-doc `dietary` + `allergens`).

**Post-join screen:** invitee lands on the new shared HomeScreen with an optional "Import your pantry into this household" banner (flow 6.4).

### 6.4 Import my pantry (invitee opt-in)

**Trigger:** Newly-joined invitee taps the banner, or finds the action later in HouseholdScreen ("Bring my stuff over").

**Operations:**
1. Confirm dialog with counts: "This adds your N pantry items, M shopping items to your household. Duplicates merge using the existing dedup rules."
2. For each `users/{inviteeUid}/inventory/*`: call `InventoryWriter.addItem(...)` against the household scope. Sprint 15.9.1's dedup + tier-sticky rule book applies — existing household tier wins on conflict.
3. Same for `shoppingItems` via `ShoppingService.addItem(...)` (matchKey dedup).
4. **Meal plan: skipped** in v1. The invitee's per-UID `users/{inviteeUid}/mealPlan/current` is preserved on disk and reappears if they leave the household, but is **unreachable through the app** while in a household — `SharedDataScope` routes meal-plan reads to `households/{hid}/mealPlan/current` (which is the owner's pre-migration plan). Snackbar after import: "Your meal plan is set aside while you're in this household. The shared plan is what you'll see — the owner can regenerate it for everyone. Your own plan returns if you leave."
5. `customItems` + `tierMemory`: merge into household versions; household wins on conflict.

### 6.5 Leave household (member, non-owner)

**Trigger:** Non-owner member in HouseholdScreen taps "Leave household". Confirm dialog ("You'll lose access to the household's inventory, shopping list, and meal plan. Your own per-user data is preserved.")

**Operations:**
1. `arrayRemove(member.uid)` on `households/{hid}.members`.
2. Update `users/{memberUid}.householdId = null`.
3. Recompute `households/{hid}/dietaryUnion/singleton` (removes leaver's dietary).
4. App routes shared-data reads back to `users/{memberUid}/*`, which still has their pre-join data.

**Edge — owner cannot leave:** if the user is the owner, the Leave button is replaced with "Delete household". Surface in copy: "Owners can't leave. Delete the household instead."

### 6.6 Kick member (owner)

**Trigger:** Owner taps a member's row in HouseholdScreen → SimpleDialog → "Remove from household". Confirm.

**Operations:** identical to 6.5 but executed by the owner against another UID. Rules permit this because the owner is updating the household doc's `members` array (member-self-update would be denied by the future `members-includes-self` rule — but member is leaving so it works either way).

**Kicked user's app state:** stays stale until next refresh. Their `users/{uid}.householdId` still says `hid`. App reads `households/{hid}/*`, gets permission-denied (no longer in `members`), detects this, **self-resets `householdId` to null locally**, and surfaces snackbar: "You've been removed from the household."

### 6.7 Delete household (owner)

**Trigger:** Owner in HouseholdScreen taps "Delete household". Strong-copy confirm dialog: "This deletes the shared inventory, shopping list, meal plan, customs, and removes all members. Each member's per-user data is preserved."

**Operations** (order matters — owner must remain in `members[]` until the final household-doc delete so subcollection deletes pass the `isMember()` rule check):
1. For each member UID in `members` **except owner**: cross-UID write to `users/{otherUid}.householdId = null` is **blocked by rules**. Solution: members self-reset on permission-denied (same as 6.6).
2. Delete all `households/{hid}/*` subcollections (client-side batched delete; for >500 docs, paginate). Owner is still in `members[]`, so the `isMember()` rule on each subcollection passes.
3. Delete `households/{hid}` doc itself (last — uses the `isOwnerOfHousehold()` rule).
4. Update `users/{ownerUid}.householdId = null`.

Members see the household disappear on their next refresh.

### 6.8 Owner deletes account (cascade)

**Trigger:** Owner uses AccountScreen → Delete Account (existing Sprint 17 flow).

**Operations:**
1. Existing `AccountService.deleteAccount()` runs (deletes `users/{uid}/*`, Firebase Auth user).
2. **New code before deletion:** if owner has `householdId`, set `households/{hid}.deletionScheduledAt = now + 7days` and surface to other members on their next refresh: "This household's owner has deleted their account. The household closes in 7 days."
3. **Cloud Function** (Sprint 17 — see §9.3): scheduled function runs daily, queries households where `deletionScheduledAt < now`, cascade-deletes subcollections + household doc.

**v1 fallback if Cloud Function deferred:** the household sits orphaned. Members can still read/write data (rules don't check `deletionScheduledAt`). After 7 days, the app blocks member writes client-side via a `deletionScheduledAt < now` check in the repository layer. Not ideal but ships. Cloud Function lands in Sprint 17, hardens this.

## 7. Pro entitlement cascade

### 7.1 The model

A member's effective Pro state is `MAX(own RC entitlement, households/{hid}.proActive)`. Whichever is more generous wins.

### 7.2 Write path (owner only)

`EntitlementService.refresh()` already computes `_tier` from RC (and the pro-tester allowlist — lines 90-93 of `entitlement_service.dart`). Append AFTER `_tier` has been set:

```dart
final user = FirebaseAuth.instance.currentUser;
final userDoc = await _db.collection('users').doc(user.uid).get();
final hid = userDoc.data()?['householdId'] as String?;
if (hid != null) {
  final household = await _db.collection('households').doc(hid).get();
  final isOwner = household.data()?['ownerUid'] == user.uid;
  if (isOwner) {
    // Use the already-computed _tier so pro-tester owners also extend Pro
    // to their household. Do NOT call PurchaseService.instance.isPro() again
    // here — that would skip the pro-tester path.
    await _db.collection('households').doc(hid).update({
      'proActive': _tier == 'pro',
      'proActiveSyncedAt': FieldValue.serverTimestamp(),
    });
  }
}
```

Fires on every refresh trigger (HomeScreen initState, paywall close, etc.). Idempotent.

### 7.3 Read path (member)

```dart
1. isProFromOwnRc = await PurchaseService.instance.isPro();             // existing
2. isProFromHousehold = false;
   if (user.householdId != null) {
     final h = await _db.collection('households').doc(hid).get();
     isProFromHousehold = h.data()?['proActive'] == true;
   }
3. _tier = (isProFromOwnRc || isProFromHousehold) ? 'pro' : 'free';
```

One extra Firestore read per refresh for household members. Acceptable.

### 7.4 Free-tier counter when `proActive == false`

Each member's existing per-UID `subscription.weeklyGenerations` counter applies independently. Each member gets their own 7/week allowance. **Households don't share a Free pool** — this matches current per-UID semantics and avoids contention.

**Note on potential abuse:** a Free household with 6 members would theoretically expose 42 free generations/week. This is mooted by §7.5 — household creation is Pro-gated, so by construction no Free household exists. If the cascade-on-Pro-expiry case occurs (owner downgrades), the household persists with Free limits per-member; pragmatically rare and not a meaningful abuse surface.

### 7.5 Household creation is Pro-gated

`EntitlementService.canCreateHousehold` getter:
```dart
bool get canCreateHousehold => isPro;
```
AccountScreen tile triggers paywall (`reason: 'household'`) if Free user taps Create.

### 7.6 Edge cases

| Scenario | Behavior |
|---|---|
| Owner has Pro, opens app, household `proActive` already true | Owner writes `proActive: true` (idempotent). |
| Owner upgrades after household exists | **Moot** — household creation is Pro-gated; owner is always Pro at household creation. Later downgrade is the relevant case. |
| Owner's Pro expires while app offline | Members continue seeing `proActive: true` until owner's next refresh. Documented limitation. Sprint 17 closes it via RC webhook + monthly sweep (§9.3). |
| Member also has their own Pro | Own RC wins. Leaving household doesn't change their Pro. |
| Member leaves while household Pro is active | `householdId` resets. Effective tier reverts to their own RC on next refresh. |
| Kicked while Pro | Same as leaving. |
| Paywall fires for member during household Pro (stale cache edge) | Member sees paywall, can complete purchase if they want. Owner's `proActive` eventually catches up. Member is Pro either way. |

## 8. Existing profiles[] integration

### 8.0 Migration — moving dietary from profile-doc to user-doc (one-time)

**Today's code state (verified against `lib/services/firestore_service.dart:76-77` + `lib/screens/profile/dietary_screen.dart:164-212`):** the auth user's dietary lives at `users/{uid}/profiles/{ownerProfileId}.dietaryRequirements` + `.allergies` (the profile flagged `isOwner: true`). `FirestoreService.saveDietary()` explicitly **strips** `dietary` and `allergies` from the user doc on write. `GeminiService` reads from `prefs.effectiveDietary`, which walks the profiles list.

**Why this matters for households:** an auth-member household design needs cross-UID dietary visibility. Two ways to achieve it:

- **(A)** Add a Firestore rules expansion that lets household co-members read each other's `users/{otherUid}/profiles/{ownerProfileId}`. Smaller code change but introduces cross-UID reads scoped to household membership.
- **(B)** Migrate the auth user's dietary fields onto the user doc itself (new fields `dietary: array<string>` and `allergens: array<string>`). Bigger code touch surface but cleaner long-term.

**16.7b commits to (B).** Reasoning: cleaner single source of truth, simpler rule surface, matches the spec's §4.1 schema as written. The one-time migration runs on household creation/join.

**Migration mechanics:**

1. On a user's first household interaction (create or join), the app reads `users/{uid}/profiles/{ownerProfileId}.dietaryRequirements` + `.allergies` and writes them to **new user-doc fields** `users/{uid}.dietary` + `users/{uid}.allergens`.
2. From that point forward, `DietaryScreen.save()` writes to **both** locations during a transition window:
   - `users/{uid}/profiles/{ownerProfileId}` — preserves solo-mode dietary union behaviour.
   - `users/{uid}.dietary` + `.allergens` — feeds the household dietary union.
3. A v1.2 cleanup task deprecates the profile-doc fields once all real users are on builds with the dual-write code path (deferred — not part of 16.7b).
4. **Solo users who never create/join a household are not migrated.** Their `DietaryScreen.save()` path is unchanged. Their dietary stays on the profile doc. They're invisible to this change.

This is the only schema migration of user data that 16.7b performs. It runs idempotently on every household create/join (cheap check: if `users/{uid}.dietary` exists, skip).

### 8.1 Two-kind member coexistence

After 16.7b:

- **Solo users** (no `householdId`): `users/{uid}/profiles/*` keeps working unchanged. Dietary union computed by walking the profiles list as today.
- **Household members:** owner's existing profiles[] migrate into `households/{hid}/profiles/*` on household creation (§6.1 step 3). The owner's own profile (`isOwner: true`) is **kept** during migration (its dietary is mirrored to the user-doc per §8.0 first, then the profile is copied to `households/{hid}/profiles/{ownerProfileId}` so dependent-only lookups still work). The household union math then reads the owner's dietary from `users/{ownerUid}.dietary` (post-migration field) AND from `households/{hid}/profiles/*.dietaryRequirements` for non-auth dependents.
- **Invitee's existing profiles[]:** stay at `users/{inviteeUid}/profiles/*` (orphaned in the household context). The invitee can `Import my pantry` (§6.4) but profiles[] is **not** part of the import in v1 (potential collision with owner's already-migrated profiles). Surface in copy: "Profiles for dependents stay personal until you opt to share them."

A v1.1+ enhancement could add "import profiles into household" with merge UX. Out of scope for 16.7b.

### 8.2 Dietary union math (post-16.7b)

```
householdDietaryUnion =
    union(users/{uid}.dietary       for uid in members[])           // new user-doc field per §8.0
  ∪ union(users/{uid}.allergens     for uid in members[])
  ∪ union(profile.dietaryRequirements for profile in households/{hid}/profiles/)
```

Cached at `households/{hid}/dietaryUnion/singleton`. Refreshed by the owner's or member's app whenever:
- An auth member edits their own user-doc dietary (DietaryScreen save — which writes to both profile-doc and user-doc during transition per §8.0).
- The owner edits a profile's dietary in HouseholdScreen.
- A member joins or leaves.
- A profile is added or removed.

The Gemini-prompt assembly path reads `households/{hid}/dietaryUnion/singleton` for household members. For solo users it walks `users/{uid}/profiles/*` as today. **Position-1 allergen preamble code path is preserved** — it just reads from a different source for household members.

**Cross-UID reads in the union recompute:** when a member's app recomputes the union after their own dietary save, they need to read the other members' `users/{otherUid}.dietary` + `.allergens`. The current `firestore.rules` blocks cross-UID user-doc reads. **Rule expansion needed:** allow `read` on `users/{uid}.dietary` and `users/{uid}.allergens` when the requester is a co-member of any household that `uid` is in. This is the **only** cross-UID user-doc read this design introduces. Trade-off: each union recompute pays N-1 cross-UID reads. Recompute is infrequent (only on save) — acceptable.

Alternative (deferred to 16.7b kickoff): each member writes their own dietary contribution to a sub-doc at `households/{hid}/memberDietary/{uid}` instead. Then the union recompute reads only inside the household scope (no cross-UID user-doc reads needed). Cleaner rules but adds a parallel write on every dietary save. Worth considering at 16.7b kickoff.

## 9. Known limitations and Sprint 17 dependencies

### 9.1 Client-side cheating

A determined attacker who decompiles the APK could write `households/{hid}.proActive: true` directly, bypassing RC.

**No new attack surface vs today:** the per-UID `subscription.weeklyGenerations` counter is already client-writable for the same reason (Sprint 17 known limitation). Both are closed together by Sprint 17 (§9.3).

### 9.2 Stale `proActive` grace period

If the owner's Pro expires while their app is offline, members continue seeing `proActive: true` until the owner reopens the app. Grace period = however long the owner stays offline.

Sprint 17's RC webhook closes this in real time. Monthly sweep is the belt-and-braces backup.

| Item | Sprint 17 fit |
|---|---|
| **RC webhook → Cloud Function → user doc + household `proActive`** | Already in Sprint 17 scope ("RC webhook"). +1.0d for the household-extension piece — needs to: (a) look up the user doc for `householdId`, (b) verify the user is owner of that household, (c) write `proActive` + `proActiveSyncedAt`. Plus edge cases: user mid-leave, household just deleted, owner UID mid-change. |

## 10. Migration

## 15. Open questions for v1.1 implementation

When Sprint 16.7b starts, these need fresh resolution (deliberately deferred from 16.7a):

1. **`members[]` array vs `members/{uid}` subcollection.** ~~Initial concern: array model can't authorise invitee self-add.~~ **Resolved in this spec via the §5.1 invitee-self-add rule branch** — the invitee includes the redeemed `joinedVia: '<code>'` field in their household update, the rule does a `get()` cross-reference against `inviteCodes/{code}` to verify `used: true && usedByUid == request.auth.uid && hid == this hid`. Member cap stays checkable via `request.resource.data.members.size() <= 6` on the household doc. **Trade-off accepted:** if two invitees race with valid codes, both can pass the rule check before either commits — net result is the second's update fails the size check (since `members.size()` is computed from `request.resource.data` at write-validation time, not at transaction-commit time). One brief race-failure for the second invitee, no cap violation. A subcollection model would have the inverse problem (no native cap rule, race produces too-many-members). The array model wins for v1.
2. **Real-time vs polling** for cross-device sync. Firestore `snapshots()` listeners on `households/{hid}/*` give true real-time. Current Elio mostly uses one-shot `get()` reads. Decision: add listeners to the shared subtree, or keep polling on screen-open? Listeners add complexity but match the "shopping list updates instantly when partner ticks an item" UX competitors ship.
3. **Member name display.** Current `users/{uid}` doesn't reliably have a display name (only the `_finishAsGuest` path doesn't set one). HouseholdScreen needs to show members by *something*. Decision: use Firebase Auth `displayName` (set on email register, may be null on Google sign-in); fall back to first segment of email; let owner rename in HouseholdScreen.
4. **Imported pantry collision policy.** Sprint 15.9.1 dedup rule is "existing tier sticky". When invitee imports their stuff, *the household's* tier is "existing" and wins. Confirm this is the desired behaviour or whether invitee should be asked.
5. **Conflict attribution.** "Kate added milk to the shopping list" attribution. Nice UX but adds a `addedBy` field to every shoppingItem write. Defer or build in v1.1?
6. **Real-time invite-code revocation UI.** Owner sees a list of active codes with "revoke" buttons? Or just rely on TTL? Lightest answer: just TTL.
7. **HouseholdScreen UI design.** Kate art direction pass.

These are implementation details, not design questions. Capture during 16.7b kickoff.
